The Problem

What is LoRA QKV?

Multi-head attention computes Q, K, V projections from input X. With LoRA (Low-Rank Adaptation), each projection adds a low-rank correction:

Q = X @ WqT + s · (X @ AqT) @ BqT
K = X @ WkT + s · (X @ AkT) @ BkT
V = X @ WvT + s · (X @ AvT) @ BvT

W = frozen base weight (large: 4096×4096)
A, B = trainable LoRA adapters (tiny: rank 8–64)
s = scaling factor

GQA (Grouped-Query Attention): Q has 32 heads → output 4096, but K/V only have 8 heads → output 1024. This 4× asymmetry matters for fusion.

Data flow — LLaMA-3 8B scale

Input X [8192, 4096] = 64 MB in bf16
Q projection Wq + LoRAq out: [8192, 4096]
K projection Wk + LoRAk out: [8192, 1024]
V projection Wv + LoRAv out: [8192, 1024]

The bottleneck: X is 64 MB. Naive implementations read it from GPU memory 6 times = 384 MB of wasted bandwidth.

A100 HBM bandwidth (the fast GPU off-chip memory): 2 TB/s. Reading X once takes ~32 μs; reading it 6× wastes ~160 μs.

Current State of the Art

How Unsloth Does It baseline

Unsloth applies matmul_lora() independently to Q, K, V. Each call makes 3 cuBLAS launches:

# Per projection (×3 for Q, K, V):
base = X @ W.t() # cuBLAS call 1: read X
XA = X @ A.t() # cuBLAS call 2: read X again
base.addmm_(XA, B.t()) # cuBLAS call 3: in-place add

3 projections × 3 calls = 9 kernel launches

X read 6 times from HBM (2× per projection for W and A)

Benchmark: 2.035 ms on A100 (rank=16, bf16)

Memory: 96 MB — tight thanks to addmm_ in-place operation

v1 — First Attempt

All-Triton Fused LoRA slower

Fuse base matmul + LoRA into one Triton kernel per projection:

Input X [8192, 4096]
Triton Q W + LoRA fused
Triton K W + LoRA fused
Triton V W + LoRA fused

Inside each kernel: output-stationary tiling with a two-pass approach:

  1. K-loop over W for base matmul: acc += X_tile @ W_tile
  2. K-loop over A for LoRA: XA_tile = X_tile @ A_tile
  3. Tiny dot for B: acc += s * XA_tile @ B_tile

Includes L2 swizzle (GROUP_SIZE_M=8) and 10 autotune configs.

v2 — The Breakthrough

Packed W+A cuBLAS + Triton Epilogue faster

Instead of separate matmuls for W and A, concatenate them and let cuBLAS read X only once:

Pack weights W_packed = [W ; A] [N+r, K]
1 cuBLAS call out = X @ W_packedT → [M, N+r] Reads X ONCE
Split base = [:, :N] XA = [:, N:]
Triton epilogue Y = base + s · XA @ BT Tiny [r, N] dot in registers

Applied 3× for Q, K, V = 6 total launches (3 cuBLAS + 3 Triton)

Why it works: LoRA's A matrix is tiny (rank × K). Packing it with W barely increases cuBLAS work, but eliminates a separate X read. The Triton epilogue's [r, BLOCK_N] dot fits entirely in registers — essentially free.

⚠ Memory: 128 MB (33% more than Unsloth's 96 MB) — packed + final output coexist during epilogue

v2_2 — Refinement

Memory Fix: In-Place Epilogue optimized

Two improvements over v2:

1. In-place epilogue

Triton writes LoRA result directly into packed output's first N columns. No separate Y allocation needed.

2. Fused 3-in-1 epilogue

Single Triton kernel handles all 3 QKV epilogues using program_id(1) to select Q/K/V.

Launches: 3 cuBLAS + 1 Triton = 4 total (down from v2's 6)

Trade-off: fused epilogue wastes GPU threads for K/V in GQA (Q is 4× larger, so K/V blocks waste ¾ of the grid). v2_3 addresses this.

v3 — Production Ready

Training Support: autograd.Function trainable

Wraps v2_3 in torch.autograd.Function with a custom backward pass:

Forward (v2_3) 1 cuBLAS + 1 Triton = 2 launches X read once, LoRA in-place
save_for_backward X, Aq, Ak, Av, Bq, Bk, Bv, Wpacked
Backward cuBLAS addmm_ chain (follows Unsloth's pattern) Gradients: dX, dAq, dBq, dAk, dBk, dAv, dBv

Correctness:

✓ torch.autograd.gradcheck passes in fp64
✓ MHA, GQA, 3D inputs, all ranks (8–64)
✓ Backward matches PyTorch reference within rtol=1e-3

Falls back to plain PyTorch for fp64 (gradcheck compatibility only — no perf impact in production bf16).

Takeaways

Key Insights

  1. 1
    "Don't compete with cuBLAS"
    Use cuBLAS for large matmuls (where it excels), Triton only for cheap fusion/epilogues. Triton matmul tops out at ~60% cuBLAS throughput for large GEMMs.
  2. 2
    "Pack weights to reduce X reads"
    Concatenating W and A (and across projections) turns 6 X reads into 1. At 64 MB per read, this saves 320 MB of HBM bandwidth.
  3. 3
    "LoRA rank is tiny — exploit it"
    The [r, BLOCK_N] dot product in the Triton epilogue fits entirely in registers. r=16 means 16 rows — essentially free compute. The LoRA addition costs almost nothing.
  1. 4
    "addmm_ isn't always optimal"
    Unsloth's addmm_ is clever (in-place), but it forces a separate cuBLAS call for X@A. A single packed GEMM + Triton epilogue is faster because it avoids re-reading X.
  2. 5
    "Memory matters — go in-place"
    v2 was faster but used 33% more memory. The in-place epilogue (v2_2+) writes directly into the packed buffer, matching Unsloth's 96 MB footprint.
  3. 6
    "Wide GEMMs don't hurt cuBLAS"
    We feared [M, 6192] output might trigger cuBLAS algorithm-selection penalties. It doesn't — v2_3 is faster than v2 (3 narrower GEMMs). Validated empirically.
