# Analysis: v2_2, v2_3, v3 — Memory Fix, Single cuBLAS, and Training Support

**Date**: 2026-05-24
**Kernel versions**: v2_2, v2_3, v3
**GPU**: NVIDIA A100-SXM4-80GB
**Config**: batch=4, seq=2048, hidden=4096, GQA 32/8 heads, bf16

## Summary

Three new versions that collectively fix v2's memory overhead, achieve up to **1.26x Unsloth** on forward, and add training support (backward pass) at **1.07-1.09x Unsloth** for fwd+bwd.

**v2_3 is the new best** — a single cuBLAS call for all QKV projections reads X from HBM only once.

## What Each Version Does

### v2_2: In-Place Epilogue + Fused 3-in-1 Epilogue

**Changes**: (1) Triton epilogue writes LoRA result in-place into packed output's first N columns (no extra Y allocation), (2) Single Triton kernel handles Q, K, V epilogues.

**Result**: Memory fixed from 128→97 MB (matching Unsloth). Speed ~1.07-1.20x Unsloth — slightly slower than v2 because the fused epilogue has wasted grid tiles for K/V in GQA (Q is 4x larger than K/V, so K/V threadblocks waste 3/4 of the grid).

**Launches**: 3 cuBLAS + 1 Triton = 4 (down from v2's 6).

### v2_3: Fully Packed QKV — Single cuBLAS Call

**Changes**: All 6 weight matrices (W_q, A_q, W_k, A_k, W_v, A_v) packed into one tall matrix. Single cuBLAS call `X @ W_all^T`, then single fused Triton epilogue applies LoRA to all 3 projections in-place.

**Result**: **New best at 1.13-1.26x Unsloth.** X is read from HBM only ONCE (vs 3x in v2, 6x in Unsloth). Saves 2 additional X reads worth of bandwidth at LLaMA-8B scale (2 × 8192 × 4096 × 2 bytes = 128 MB bandwidth saved).

**Launches**: 1 cuBLAS + 1 Triton = **2 total** (vs v2's 6, Unsloth's 9).

### v3: autograd.Function with Backward Pass

**Changes**: Wraps v2_3 forward in `torch.autograd.Function`. Backward uses cuBLAS `addmm_` chain following Unsloth's pattern. Falls back to plain PyTorch for fp64 (gradcheck).

**Result**: Forward matches v2_3 speed. **Full fwd+bwd is 1.07-1.09x Unsloth.** Gradcheck passes in fp64 for MHA, GQA, 3D inputs, and all ranks.

**Why backward beats Unsloth**: v3's forward reads X once (vs Unsloth's 6 reads). The backward is identical to Unsloth's addmm_ pattern, so the net gain comes entirely from the faster forward.

## Remaining Improvement Opportunities

1. **Backward fusion**: v3's backward still uses separate cuBLAS calls. Could pack dX computation similar to forward.
2. **Hopper/Blackwell**: CODA-style epilogue programs could fuse the Triton epilogue into the cuBLAS GEMM's epilogue stage, eliminating the second launch entirely.
3. **cuBLAS algorithm**: The wide output ([M, 6192]) doesn't appear to suffer the algorithm-selection penalty we feared — v2_3 is faster, not slower. This validates the packing approach.
4. **Contiguous output path**: Currently returns views into the packed buffer. For workloads that need contiguous Q/K/V, a copy is needed downstream. Could provide an option to pre-allocate contiguous outputs and copy within the kernel.
